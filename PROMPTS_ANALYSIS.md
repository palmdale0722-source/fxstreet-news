# G8 货币强弱矩阵、刺客精选、货币驱动力 - 完整提示词分析

## 概述

FXStreet 系统中的 G8 货币强弱矩阵由三个主要 AI 调用组成：
1. **货币强弱评分** - 对 8 种 G8 货币进行三层驱动力评分
2. **刺客精选** - 基于评分差异，精选强弱差最大的货币对
3. **经济总结** - 为每个国家生成经济状况总结

---

## 1. 货币强弱评分提示词

### 系统角色定义
```
你是一位专业的外汇基本面分析师，精通《外汇交易三部曲》中的"逻辑层次分析矩阵"方法。
```

### 输入数据结构

系统会发送以下四类数据：

#### 1.1 经济指标数据（来自 TradingEconomics）
```
【美国（USD）】
  - 失业率: 3.8% (前值: 3.9%)
  - 非农就业人数: 275K (前值: 300K)
  - CPI 同比: 3.2% (前值: 3.4%)
  - 核心 CPI: 3.8% (前值: 3.9%)
  ...（每个国家约 15 个关键指标）

【欧元区（EUR）】
  - 失业率: 6.2% (前值: 6.3%)
  - PMI 制造业: 45.2 (前值: 44.8%)
  ...
```

**数据来源**：每个国家保留 30 个最新经济指标，包括：
- 就业数据（失业率、非农就业等）
- 通胀数据（CPI、核心 CPI、PPI）
- 增长数据（GDP、工业生产、零售销售）
- 制造业 PMI、服务业 PMI

#### 1.2 央行货币政策动态（来自 FXStreet）
```
【FED（USD）】
  - Federal Reserve 维持利率在 5.25%-5.50%
  - Powell：通胀仍未回到 2% 目标，需要更多时间
  - 市场预期 Q2 首次降息

【ECB（EUR）】
  - ECB 维持利率在 4.25%
  - Lagarde：通胀粘性强于预期，暂无降息时间表
  ...
```

**数据来源**：每个央行最新 5 条新闻标题

#### 1.3 FXStreet 最新新闻
```
- 美联储鹰派声明支撑美元
- 欧元区经济数据疲弱，ECB 面临降息压力
- 日本央行暗示加息时间表
...
```

#### 1.4 FXStreet 分析文章
```
- 美元强势背后的三大支撑因素
- 欧元下行风险评估
...
```

### 评分体系（三层权重分析）

#### 顶层（地缘与制度）权重 30%
```
+3: 极强避险需求流入 / 政治稳定 / 制度优势显著
+1~+2: 轻微避险偏好 / 政治稳定
0: 中性
-1~-2: 轻微地缘风险 / 政治不确定性
-3: 严重地缘危机 / 政治动荡 / 制度风险
```

#### 中层（货币政策/利差）权重 40%（核心层）
```
+3: 明确鹰派，加息周期，利差优势显著
+1~+2: 偏鹰派，维持高利率，利差有支撑
0: 中性，观望
-1~-2: 偏鸽派，降息预期，利差收窄
-3: 明确鸽派，降息周期，利差劣势显著
```

#### 底层（具体数据脉冲）权重 30%
```
+3: 多项核心数据大幅好于预期（非农、PMI、通胀等）
+1~+2: 数据偏强，部分好于预期
0: 数据符合预期，无明显脉冲
-1~-2: 数据偏弱，部分差于预期
-3: 多项核心数据大幅差于预期
```

**综合评分公式**：
```
compositeScore = topLayerScore * 0.3 + midLayerScore * 0.4 + bottomLayerScore * 0.3
范围：-3 到 +3
```

```json
{
  "currencies": [
    {

      "topLayer": {
        "score": 1.5,
        "summary": "顶层驱动简要说明（50字以内）",

      },
      "drivingAnalysis": {
        "direction": "偏多，建议做多USD（100字以内）",
        "rhythm": "慢趋势（解释原因，50字以内）",

      },
```

## 2. 刺客精选提示词

### 系统角色定义
```
你是一位专业的外汇基本面分析师，精通"刺客原则"——只狙击强弱差最大、逻辑最清晰的货币对。
```

### 输入数据

#### 2.1 当前 G8 货币强弱评分（来自上一步输出）
```
🇺🇸 USD: 2.15分 (顶层1.5 + 中层2.0 + 底层0.5)
🇪🇺 EUR: -0.85分 (顶层0.0 + 中层-1.0 + 底层-0.55)
🇯🇵 JPY: 1.25分 (顶层1.0 + 中层1.5 + 底层0.5)
🇬🇧 GBP: 0.35分 (顶层0.5 + 中层0.2 + 底层0.3)
🇦🇺 AUD: -0.45分 (顶层-0.2 + 中层-0.5 + 底层-0.4)
🇳🇿 NZD: -0.65分 (顶层-0.3 + 中层-0.8 + 底层-0.5)
🇨🇦 CAD: 0.15分 (顶层0.2 + 中层0.0 + 底层0.2)
🇨🇭 CHF: 1.85分 (顶层1.5 + 中层1.8 + 底层2.0)
```

#### 2.2 强弱差最大的候选货币对（自动计算）
```
USD/EUR: 强弱差 3.00
CHF/EUR: 强弱差 2.70
USD/AUD: 强弱差 2.60
JPY/EUR: 强弱差 2.10
CHF/NZD: 强弱差 2.50
```

#### 2.3 参考数据
- 经济指标（前 3000 字符）
- 央行政策动态（前 2000 字符）

### 刺客原则

```
当强弱差（综合评分差的绝对值）> 3 时，标记为"高胜率机会"
```

### 每个货币对的分析结构

#### 2.3.1 逻辑层次拆解
```
1. 顶层驱动（大环境）：当前最重要的地缘/制度因素
2. 中层动力（央行逻辑）：两国央行政策分化的核心逻辑
3. 底层脉冲（近期数据）：最近影响走势的具体数据
```

#### 2.3.2 驱动分析三问
```
1. 方向：基于强弱差给出的多空建议
2. 节奏：是"慢趋势"（基本面驱动，周/月级别）还是"快波动"（预期差导致，日/小时级别）
3. 失效点：明确什么基本面事件发生会导致该驱动逻辑作废
```

#### 2.3.3 预期差监控
```
1. 当前市场普遍预期值
2. 实际驱动力方向
3. 两者之间的"剪刀差"描述
```

## 4. 分段发送方案 - 避免 503 错误

### 问题分析

**原因**：单次请求包含 23KB+ 的数据，可能导致：
1. 请求超时（LLM API 处理时间过长）
2. Token 限制（某些 API 有单次请求的 token 上限）
3. API 过载（并发请求过多）

### 解决方案：分段更新策略

#### 方案 A：分货币评分（推荐）

**第 1 步**：分组评分（2 个 API 调用）

```
调用 1：评分前 4 个货币（USD, EUR, JPY, GBP）
- 输入：4 国经济指标 + 央行新闻 + FX 新闻
- 输出：4 个货币的三层评分

调用 2：评分后 4 个货币（AUD, NZD, CAD, CHF）
- 输入：4 国经济指标 + 央行新闻 + FX 新闻
- 输出：4 个货币的三层评分
```

**优点**：
- 每次请求大小减半（~12KB）
- 降低超时风险
- 可并行执行

**缺点**：
- 需要 2 次 API 调用（成本翻倍）
- 刺客精选需要等待两次调用完成后再执行

#### 方案 B：分数据源评分（高级）

**第 1 步**：仅用央行政策 + 利差数据评分

```
调用 1：基于央行政策的中层评分
- 输入：8 央行新闻 + 利率数据
- 输出：8 个货币的中层评分（权重 40%）
```

**第 2 步**：基于经济数据的底层评分

```
调用 2：基于经济指标的底层评分
- 输入：8 国经济指标（去掉央行新闻）
- 输出：8 个货币的底层评分（权重 30%）
```

**第 3 步**：基于地缘政治的顶层评分

```
调用 3：基于地缘政治的顶层评分
- 输入：FX 新闻 + 分析文章
- 输出：8 个货币的顶层评分（权重 30%）
```

**优点**：
- 每次请求大小最小（~5KB）
- 最大化并行性
- 可独立缓存各层评分

**缺点**：
- 需要 3 次 API 调用（成本最高）
- 实现复杂度高

#### 方案 C：缓存 + 增量更新（最优）

**第 1 步**：首次完整评分（1 次调用）

```
调用 1：完整 G8 评分
- 缓存结果 6 小时
```

**第 2 步**：增量更新（仅更新变化的数据）

```
仅当以下数据发生变化时触发更新：
- 央行政策公告（直接更新中层评分）
- 关键经济数据发布（直接更新底层评分）
- 重大地缘事件（直接更新顶层评分）
```

**优点**：
- 大幅降低 API 调用频率
- 成本最低
- 系统最稳定

**缺点**：
- 需要事件驱动架构
- 实现复杂度最高

## 5. 推荐实施方案

### 短期（立即）：方案 A - 分货币评分

**实施步骤**：

1. 修改 `buildCurrencyStrengthPrompt` 支持货币列表参数
2. 拆分为两次调用：
   ```typescript
   const scores1 = await invokeLLM({
     messages: [{
       role: "user",
       content: buildCurrencyStrengthPrompt(
         economicData.filter(d => ["USD", "EUR", "JPY", "GBP"].includes(d.currency)),
         centralBankNews,
         fxstreetNews,
         analysisArticles
       )
     }]
   });
   
   const scores2 = await invokeLLM({
     messages: [{
       role: "user",
       content: buildCurrencyStrengthPrompt(
         economicData.filter(d => ["AUD", "NZD", "CAD", "CHF"].includes(d.currency)),
         centralBankNews,
         fxstreetNews,
         analysisArticles
       )
     }]
   });
   ```

3. 合并结果
4. 执行刺客精选（1 次调用）

**预期效果**：
- 请求大小：23KB → 12KB（减少 48%）
- 超时风险：大幅降低
- 成本增加：100%（从 2 次调用变为 3 次）

### 中期（1-2 周）：方案 C - 缓存 + 增量更新

**实施步骤**：

1. 新增 `currency_strength_cache_metadata` 表
   ```sql
   CREATE TABLE currency_strength_cache_metadata (
     id INT PRIMARY KEY,
     lastFullUpdate DATETIME,
     lastCentralBankUpdate DATETIME,
     lastEconomicDataUpdate DATETIME,
     lastGeopoliticalUpdate DATETIME,
     cacheValidUntil DATETIME
   );
   ```

2. 实现事件驱动更新
   ```typescript
   // 央行政策变化 → 更新中层评分
   // 经济数据发布 → 更新底层评分
   // 地缘事件 → 更新顶层评分
   ```

3. 缓存策略
   - 完整评分缓存 6 小时
   - 增量更新缓存 1 小时
   - 超过缓存时间自动触发完整更新

## 6. 成本估算

### 当前方案（单次完整调用）
- 每小时 1 次：每月 ~720 次调用
- 平均 token 数：~3000 tokens
- **月成本**：取决于 LLM 定价

### 方案 A（分货币评分）
- 每小时 3 次（2 次评分 + 1 次刺客精选）：每月 ~2160 次调用
- 平均 token 数：~1500 tokens（降低 50%）
- **月成本**：约增加 50%（调用次数翻倍，但单次 token 减半）

### 方案 C（缓存 + 增量更新）
- 完整评分：每天 1 次（24 次/月）
- 增量更新：平均每天 2 次（60 次/月）
- 总计：~84 次调用/月
- **月成本**：约降低 85%

## 7. 建议

1. **立即采用方案 A**：分货币评分，降低单次请求大小，解决 503 超时问题
2. **监控成本**：如果成本增加过多，推进方案 C 的实施
3. **优化数据**：
