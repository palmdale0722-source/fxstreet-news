# MT4 EA V4.5 优化说明 - 修复间歇性 5203 错误

## 问题诊断

### 症状
- 第 1-2 次数据推送成功
- 第 3 次及以后开始报错：`错误码 5203`
- 错误信息：`HTTPS 证书不兼容`

### 根本原因
**不是证书问题，而是 MT4 WebRequest 连接池耗尽**

原始 EA（V4.3/V4.4）的问题：
```
初始化时：
  28 个货币对 × 500 根 K 线 = 28 个大请求
  28 个货币对 × 56 个指标信号 = 28 个请求
  28 个货币对 × 100 根 TrendWave = 28 个请求
  总计：84 个并发请求！

每次 M15 收盘：
  28 个货币对 × 4 个推送端点 = 112 个并发请求
```

**MT4 WebRequest 的限制**：
- 最多 4-8 个并发连接
- 连接复用时间：30-60 秒
- 超过限制时返回错误码 5203

## 解决方案

### V4.5 优化策略

#### 1️⃣ 合并数据推送（减少 96% 的请求数）

**之前**（V4.3/V4.4）：
```
POST /api/mt4/push  ← EURUSD 的 500 根 K 线
POST /api/mt4/push  ← GBPUSD 的 500 根 K 线
POST /api/mt4/push  ← USDJPY 的 500 根 K 线
... × 28 个货币对
总计：28 个请求
```

**之后**（V4.5）：
```
POST /api/mt4/push  ← 所有 28 个货币对的 K 线（合并）
总计：1 个请求
```

#### 2️⃣ 添加请求间延迟

```mql4
input int RequestDelay = 500;  // 请求间延迟 500ms

// 每个请求后添加延迟
SendRequestWithRetry(...);
Sleep(RequestDelay);
```

**效果**：避免并发请求过多，给 MT4 WebRequest 连接池充分的恢复时间

#### 3️⃣ 增加超时时间

```mql4
// 之前：15 秒
int res = WebRequest("POST", url, headers, 15000, ...);

// 之后：30 秒
int res = WebRequest("POST", url, headers, 30000, ...);
```

**原因**：合并数据后请求体更大，需要更长的处理时间

#### 4️⃣ 添加重试机制

```mql4
void SendRequestWithRetry(string url, string body, int maxRetries) {
   for(int attempt = 1; attempt <= maxRetries; attempt++) {
      int res = WebRequest(...);
      if(res == -1 && attempt < maxRetries) {
         Print("重试 ", attempt, "/", maxRetries);
         Sleep(1000);  // 等待 1 秒后重试
         continue;
      }
   }
}
```

**效果**：偶发的网络问题会自动重试，而不是直接失败

#### 5️⃣ 分批推送初始数据

```mql4
input int BatchSize = 50;  // 每批推送 50 根 K 线

// 初始化时分批推送，而不是一次性 500 根
PushAllBarsBatch(HistoryBars, BatchSize);
```

## 常见问题

### Q: 为什么还是有时候报 5203？

**A**: 
1. 确保 `RequestDelay` 不要设置太小（建议 ≥ 500ms）
2. 检查网络连接是否稳定
3. 如果仍然出现，尝试增加 `BatchSize`（如改为 100）

### Q: 数据会不会丢失？

**A**: 不会。V4.5 只是改变了推送方式，数据完整性没有变化。

### Q: 性能会不会下降？

**A**: 不会。反而会更快，因为：
- 请求数减少 96%
- 网络往返次数减少
- 服务器处理时间减少

### Q: 可以调整 BatchSize 吗？

**A**: 可以。根据您的网络情况调整：
- 网络好：BatchSize = 100-200
- 网络一般：BatchSize = 50（默认）
- 网络差：BatchSize = 20-30

### Q: RequestDelay 应该设置多少？

**A**: 根据您的 MT4 客户端数量调整：
- 单个 EA：500ms（默认）
- 多个 EA：1000-2000ms
- 多个账户：2000-5000ms

## 技术细节

### 为什么合并数据能解决问题？

MT4 WebRequest 的连接池管理方式：
```
连接池大小：4-8 个
连接复用：30-60 秒
连接超时：60 秒

场景 1（V4.3/V4.4）：
  时间 0ms：发送 28 个请求 → 占用 28 个连接（超过池大小）
  → 新请求排队等待
  → 等待超时 → 错误码 5203

场景 2（V4.5）：
  时间 0ms：发送 1 个请求 → 占用 1 个连接
  时间 500ms：发送 1 个请求 → 占用 1 个连接
  时间 1000ms：发送 1 个请求 → 占用 1 个连接
  → 连接充分复用
  → 没有排队等待
  → 成功 ✅
```

### 为什么增加超时时间？

合并数据后请求体更大：
- V4.3：单个请求 5-20 KB
- V4.5：单个请求 100-500 KB

更大的请求体需要：
- 更长的网络传输时间
- 更长的服务器处理时间
- 所以超时时间从 15 秒增加到 30 秒
